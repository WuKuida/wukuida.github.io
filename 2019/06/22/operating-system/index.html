2.2 操作系统的系统调用，异常，中断

系统调用：应用程序主动向操作系统发出服务请求
异常：（来源于不良的应用程序）非法指令或者其它坏的处理状态，使得操作系统不得不进行处理
中断：来自不同的硬件设备的计时器和网络的中断

中断处理过程

  1. 保存当前处理状态
  2. 中断服务程序处理
  3. 清楚中断标记（由硬件产生）
  4. 恢复之前保存的处理状态

异常处理流程

  1. 保存现场
  2. 异常处理
    • 处理方式一：杀死产生异常的服务
    • 处理方式二：修复产生异常的原因（如果这个原因是操作系统造成的话），重新执行
  3. 恢复现场

系统调用处理流程：

以标准C库为例

  1. 应用程序调用printf（）时
  2. 直接或间接的通过lib库调用write（），
  3. 此时会cpu从用户态到内核态，应用程序堆栈切换到内核堆栈
  4. 在执行write的时候，操作系统可能会访问相应的硬件，比如屏幕
  5. 系统调用处理结束返回成功或失败
  6. 应用程序获取系统调用的结果之后，继续执行（当然应用程序可能调用之后就去执行别的了，不用等系统调用返回结果）

3.3 连续内存分配：内存碎片与分区动态分配

内存碎片：

  • 外部碎片：在分配单元间的未使用内存
  • 内存碎片：在分配单元中的未使用内存

内存分配算法：

  • 首次适配算法：按地址从低到高找到第一块满足需求的内存块
    • 简单（优点）
    • 易于保留较大的地址空间（优点）
    • 容易产生外部碎片（缺点）
    • 不确定性（缺点）
  • 最佳适配：找到一个刚好大于需求的空闲内存块
    • 最小化外部碎片产生的尺寸（优点）
    • 比较适合较小需求的内存块分配（优点）
    • 简单（优点）
    • 容易产生外部碎片（缺点）
    • 分配慢（缺点）
    • 容易产生的过小的外部碎片，不好用（缺点）
  • 最差适配：找到最大的一个空闲内存块来满足需求
    • 比较适合中等尺寸需求的内存块分配（优点）
    • 重分配慢（缺点）
    • 较多外部碎片（缺点）
    • 会破坏大块的，容易导致大块的需求不能满足（缺点）

3.4 连续内存分配：压缩式与交换式碎片整理

压缩式：
将分配的内存压缩到一起，去掉中间的碎片空间
注意点：

  • 要在程序停止的时候做这个
  • 要注意内存拷贝的开销

交换式：
将内存中应用程序（数据）暂时放到硬盘（虚拟内存）中
注意点：

  • 要选择哪个程序放到硬盘（虚拟内存）中
  • 要在什么时候进行这个操作

非连续分配的优点：

  • 一个程序的物理地址空间是非连续的
  • 更好的内存利用和管理，减少碎片
  • 允许共享代码和数据（共享库等…)
  • 支持动态加载和动态链接（不懂）

非连续分配的缺点：

  • 如何建立虚拟内存和物理地址之间的转换

4.1 非连续内存分配：分段

  • 可以按一下几个模块分段

    1 堆

    2 栈

    3 程序数据

    4 程序text段（包括库和用户代码）

  • 段地址访问机制，程序访问内存地址需要一个二维的二元组（s，addr）

    1 s是段地址

    2 addr是段内偏移

  • 通过存储在cpu内MMU的段表（seg table),由操作系统维护

    1 逻辑地址的s，就是段表的index

    2 段表那存了每段的起始物理地址base，和每段的长度limit，如果逻辑地址的段内偏移大于该段的长度，cpu就会产生内存异常报错

4.2 非连续内存分配：分页

基本概念：

  • 划分物理内存至大小固定的帧（块）frame
  • 划分逻辑内存至大小固定的页 page
  • 建立逻辑地址到物理地址的转换
    • 页表（由操作系统建立，存在内存中）
    • MMU中的快表（TLB）

逻辑地址到物理地址映射过程：

  1. 对于一个逻辑地址（page_num, offset)
  2. 通过页表（实际就是一维数组，它的页表基址寄存器（PTBR）是操作系统设置的），将page_num转化成frame_num(frame_num = tabel[page_num+PTBR])
  3. 得到（frame_num, offset)
  4. frame_num * 帧（块，页）大小 + offset 就是物理地址

4.3 非连续内存分配：页表-概述

每个运行的程序都会有一个页表，属于程序运行状态，会动态变化

页表的结构：

  • PTBR:页表基址寄存器
  • table[page_num + PTBR]并不只有frame_num
    • Flags(标识位,前面的几个bit），是否写过等
      • dirty bit
      • resident bit（判断是否存在相应物理地址）
      • clock/reference bit
    • frame_num:帧号

时间问题：因为页表空间比较大，所以放到内存中，所以逻辑地址到物理地址需要访问两次内存。
解决方案：通过cpu中的MMU的TLB快表缓存（Caching）

- TLB使用associative memory（关联内存），实际上就是map，(key, value) = (page_num, frame_num)
- 如果TLB命中，逻辑页号对应的物理页号可以直接获取
- 如果TLB没有命中，将对应的映射关系存入TLB


空间问题：对于64位的操作系统，如果帧的大小是1KB，那么页表（数组）的长度应该为$2^{64-10}$,如果每个应用都有对应的页表，那就更大了
解决方案：通过间接访问（多级也表）4.4 页表-多级

- 将page_num分成page_num1和page_num2
- 将page_num1通过一级也表，找到�二级也表的基址（base),然后将page_num2映射到物理帧号
- 因为访问一级页表里面，可能就不存在对应的二级页表，所以节省了空间

5.1 虚拟内存的起因

内存不够用
最理想的内存，容量大，速度快，能保存数据

5.2 覆盖技术

目标：
在较小的内存中运行较大的程序。

实现：
按程序划分成若干个相对独立的模块，让那些不会同时执行的模块共享同一块内存区域。

  • 必要部分（常用功能）的代码和数据要常驻内存
  • 可选部分（不常用功能），平时放在外存（磁盘）中，需要时加载到内存
  • 不存在调用关系的模块不必同时存在内存中，故他们可以共享同一块内存空间

优点：
可以在相对比较小的内存中跑比较大的程序

缺点：

  • 太麻烦了，要程序模块间的覆盖要程序员自己设计
  • 从外存（磁盘）加载到内存，费时

5.2 交换技术

目标：
多道程序在内存中的时候，让需要运行的程序或将要运行的程序获得更多的内存资源

实现：

  • 将暂时不用的程序从内存移动到外存中（swap out)
  • 将需要运行但是放在外存中的程序移入到内存中（swap in)

考虑的问题：

  • 需要考虑换入换出的时间（只有当有程序内存不够用时才进行）
  • 交换区的大小（感觉就是虚拟内存的概念）极端情况只有一个程序留在内存里，别的都放到交换区
  • 换出的程序，换入之后可能不在原来的地址空间

和覆盖技术的区别：

  • 覆盖技术

    • 颗粒度是程序内的一个模块
    • 需要程序员自己设计（太麻烦了）
    • 对正在运行中的程序进行操作

  • 交换技术

    • 颗粒度是一整个程序（太浪费时间了）
    • 由操作系统完成
    • 对没有正在运行的程序操作

5.3 虚存技术（上）

目标：

  • 像覆盖技术那样，只把程序的部分内容放在内存中，�且无需人为干预
  • 像交换技术那样，能够实现进程在内存与外存之间的交换

程序的局部性原理：

  • 时间局部性：一条指令（数据）一次访问和下一次访问都集中在较短的时间
  • 空间局部性：当前指令（数据）和下一条指令（数据）都集中再较小的一块区域

基本特征：

  • 大的用户空间：因为把磁盘也当成内存使用
  • 部分交换：与交换技术相比，虚拟存储的换入和换出的内存是对部分虚拟地址空间进行的
  • 不连续性：物理内存分配的不连续，虚拟地址空间使用的不连续

5.4 虚存技术（下）

虚拟页式内存管理：

  • 大部分虚拟存储系统都采用虚拟页式存储管理结构：即在页面存储管理的基础上，增加请求调页和页面置换功能
  • 基本思路：
    • 将程序载入到内存中的时候，只将该程序必要的一部分先载入内存中
    • 在运行过程中，如果发现要运行的程序或者需要访问的数据不在内存中，则向操作系统发出缺页中断请求，让操作系统将外存中的页面载入内存

为了让有些页面常驻内存还有锁定位（lock bit)，这些页面不在页面置换算法的处理范围内。

缺页中断处理流程：

  1. 如果内存中间中有空闲的物理页面，则分配物理页面f，然后跳到4
  2. 采用某分页置换算法，找到需要被替换的物理页f，该物理页对应的逻辑页是p。如果该页被修改过则将它写回到外存。
  3. 将p所对应的页表项进行修改，驻留位改成0
  4. 将需要访问的页面p2载入到物理页f中
  5. 将p2对应的也表项的驻留位改成1，无力页号改成f
  6. 重新运行被中断的指令

如果是运行过程中产生的数据，在外存中本来页没有，这种是怎么处理的？？？因为我感觉在内存新产生的数据再外存中是没有的，流程上应该稍微有一点不一样吧？？？

后备存储（Backing Store）：
如何保存未被映射的页：

  • 能够简单地识别在�二级存储器中的页
  • 交换空间（磁盘或者文件）：特殊格式，用于存储未被映射的页面

后备存储（backing store)概念：

  • 一个虚拟地址空间的页面可以被映射到一个文件（在二级存储中的某个位置）–数据文件
  • 代码段：映射到可执行的二进制文件
  • 动态加载的共享库程序段：映射到动态调用的库文件
  • 其他段：可能被映射到交换文件（swap file)–运行过程中产生的数据

6.1 最优页面置换算法(OPT:optimal)

功能：
当缺页中断产生的时候，需要调入新的页面但是内存已满，需要选择内存中换出的页面

目标：
尽量减少缺页中断的次数

最优页面置换算法：
当一个缺页中断产生的时候，找到内存中离下一次访问最久的那个页面，把它替换出去（最理想的，不可能实现）

6.2 先进先出页面置换算法（FIFO：First-in First-out）

基本思路：

选择内存中驻留时间最长的页面并淘汰掉，相当于一个队列（链表），先进先出。

优点：

只需要一个指针

缺点：

忽略了程序运行的时空局部性，经常调出经常访问到的页，并且有Belady现象

6.3 最近最久未使用算法（LRU:Least Recently Used)

基本思路：

选择内存中最久未被使用的那个页面淘汰

优点：

相对比较接近最优页面置换算法，也利用了程序运行的时空局部性

缺点：

因为需要记录每个页面的使用时间的先后次序。假如用的是链表：需要用到两个指针。如果访问了在内存中存在的页面，则需要在链表中找到这项把这一项从链表中移到链表的头部。如果不存在，则淘汰链表尾部的元素，将新加入的放入链表头

6.4 时钟页面置换算法（Clock）

基本思路：

  • 如果一个物理页面最近被访问了（读/写），那么在页表中对应的那个页表项内的访问位（access bit)会被设置成1

  • 将存在内存中的页面的页表项组成一个环形链表（类似一个时钟）

  • 当发生缺页中断的时候，判断当前指针指向的页表项中的访问位是否位1，是1的话设置成0，指针往后挪一位。最先找到一个访问位是0的，则就是我们需要调出去的页面

  • 找到需要调出内存中的页之后，还是一样的操作，将该页表项中的驻留位设置成0，如果这个页被写过，还需要同步到外存中。然后将我们要加入的逻辑页对应的页表项的物理地址改成刚刚腾出来的，外存的数据写到该物理页，驻留位改成1，访问位设置成1。

  • 指针往后挪一个,不挪的话，在极端情况下刚刚加入的时候，恰巧其它页表项的访问位也都是1，那么下一次还是先淘汰了这个新加入的页，感觉不太合理了。

6.5 二次机会法

基本思路：

  • 和时钟页面置换算法差不多，只不过用了dirty bit

  • 当access bit是0时，看dirty bit是不是1，是的话，从1变成0。

  • 如果access bit和dirty bit都是0，则换出

如果dirty bit变成0了，我们怎么知道这个位是修改过的呢，当想要把这个页替换出去的时候，怎么判断要不要对外存中的数据进行修改？

6.6 最不常用算法（LFU：Least Frequently Used)

基本思路：

  • 当缺页中断发生的时候，将访问次数最少的页替换出去

实现方法：

  • 对每一个页面都设置一个访问计数器，当一个页面被访问的时候，计数器+1。

  • 当要淘汰一个页面的时候，找到计数最少的那个页面

缺点：

  • 计数器加一，硬件成本太高

  • 维护，查找计数最少的页，效率太低

  • 而且对刚加入内存的页不友好

和LRU的区别：

  • LRU考虑的是 保留最近访问的

  • LFU考虑的是保留访问频次高的

6.7 Belady现象，FIFO,LRU,Clock的比较

belady现象：在采用FIFO的时候，增加分配的物理内存页，反而会造成更多的缺页中断

belady现象原因：FIFO与进程访问内存的时空局部性相矛盾

  • 观察就可以发现对于size=4的，因为在第5，6次访问的时候1和2都还在所以没有动，但是当第7次访问5时，却把最近访问过的1淘汰了

  • 相对的，当size=3时，第5，6次访问的时候把1和2重新加入了，在第七次访问的时候并没有把最近访问过的1和2淘汰掉

其它：

  • LRU不存在Belady现象，因为越多的物理内存，那么保存页面肯定会包含较少的物理内存保存页面

  • Clock也不存在Belady现象，因为当只有n个物理内存的时候，如果某次缺页中断要淘汰a页，也就是其它n-1页都比a页有价值。那么同样的情况下，当有n+1个物理内存的时候，要不淘汰a页，要么a页比新多出来的那一页有价值，淘汰多出来的那一页。所以后者肯定包含前者。但是改变了指针的访问顺序，不知道这个要怎么说明。

相同点：

  • FIFO，LRU，Clock都是局部页替换算法，对一个正在运行的程序，决定要替换出去的页

不同点：

  • FIFO:是根据页面进入内存的时间排序

  • LRU:是根据页面访问时间进行排序，如果每次都要加入新的页面，那么页面访问时间就等于页面进入内存的时间，LRU就会退化成FIFO

  • Clock:是相对LRU的简化版

6.8 局部替换算法的问题，工作集模型

局部页面置换算法的问题：

  • 局部页面置换算法比较依赖于分配这个进程的内存空间

  • 程序运行过程中，程序对内存的需求是变动的

工作集：一个进程当前最近一段时间使用的逻辑页面集合-

  • t是当前执行的时刻

  • Δ 是工作集窗口大小（working-set window)，就好一个时间窗口

  • W(t, Δ) = 当前时刻t往前Δ的时间范围内，访问的逻辑页面组成的集合

  • |W(t, Δ)|集合内元素的个数

常住集:当前时刻，进程中实际驻留在内存中的页面集合

  • 工作集是进程运行的固有属性

  • 常驻集取决于工作集，系统分配给进程的内存空间，采用的页面置换算法

  • 当分配的内存空间足够是，常驻集包含工作集，这个时候程序运行相对会比较顺利，不会有太多缺页中断（直到工作集发生变化，过渡到另一个稳定状态

  • 当分配的内存空间足够之后，再增加并不会明显改善缺页中断

6.9 两个全局置换算法

工作集页置换算法：

  • 在内存中页面如果不在工作集中了，那么也会被调出内存

  • 我理解如果在前面的条件下，分配给一个进程的物理内存不够，发生缺页中断，还是需要一个局部页面替换算法，这个工作集替换算法只是让调出页面的途径多了一种

基于缺页率的页面置换算法：

缺页率（缺页次数/内存访问次数）的因素有以下几点：

  • 页面置换算法

  • 分配给进程的物理内存

  • 页面大小

  • 程序的编写方式

基本思路：

  • 常驻集可变，本质就是给各个运行中的进程动态分配物理内存

  • 具体实现，可以使用缺页率算法（PFF，page falut frequency)

    • 设置一个阈值T

    • 如果两次缺页中断时间间隔<=T,那么就增加物理内存，让新增的页直接保留

    • 如果两次缺页中断时间间隔>T,那么就减少物理内存，只保留两次缺页中断中访问过的页

7.1 进程的定义

进程的定义：

  • 一个具有一定独立功能的程序在一个数据集合上的一次动态执行过程

  • 上面的太专业了，我就只会说程序运行的时候就变成进程了

7.2 进程的组成

进程的组成：

  • 程序的代码
  • 程序处理的数据

  • 程序计数器中的值，指示下一条将运行的指令

  • 一组通用的寄存器的当前值，堆，栈

  • 一组系统资源（如打开的文件）

总之，进程包含了正在运行的一个程序的所有状态信息

进程和程序的联系：

  • 程序是产生进程的基础

  • 程序的每次运行都会产生不同的功能

  • 进程是程序功能的体现

  • 通过多次执行，一个程序可以对应多个进程

  • 通过调用关系，一个进程可以包括多个程序

进程与程序的区别：

  • 进程是动态的，程序是静态的；进程是程序的执行，进程有核心态/用户态

  • 进程是暂时的，程序是永久的；进程是一个状态变化的过程，程序可以长久保存

  • 进程与程序的组成不过：程序是有序代码段的集合；进程的组成包括程序，数据和进程控制块

7.4 进程控制结构

进程控制块（PCB）：

  • 操作系统管理控制进程运行所用的信息集合

  • PCB是进程的唯一标志，PCB存在才意味着进程存在

PCB的内容：

  • 进程标示信息：pid，以及父进程pid，创建的用户

  • 处理机状态信息：保存进程的运行状态的现场信息

    • 用户可见寄存器：用户程序可以使用的数据，地址的寄存器

    • 控制和状态寄存器：如程序计数器（PC），程序状态字（PSW）

    • 栈指针：过程调用/系统调用/中断处理和返回时需要用到它

  • 进程控制信息：

    • 调度和状态信息：用于调度系统调度进程并占用处理机使用

    • 进程间通信信息：为支持进程间通信的各种标示，信号，信件等，这些信息存在接收方的PCB中

    • 存储管理信息：包含指向本进程映像存储空间的数据结构

    • 进程所用资源：说明由进程打开，使用的系统资源，如打开的文件。

    • 有关数据结构连接信息：进程可以连接到一个进程队列中，或连接到相关的其它进程PCB

PCB的组织方式：

  • 链表：同一状态的进程PCB成一链表（就绪链表，阻塞链表）

  • 索引表：不常用，略

7.6 进程状态变化模型

进程的三种基本状态：

  • 运行状态（Running):正在占用cpu资源

  • 就绪状态（Ready):获得了除cpu以外的所有资源

  • 等待状态（Blocked）:一个进程因为需要等待某一个事件而暂停运行。如IO等待。

7.7 进程挂起

进程挂起意味着进程没有在内存，在磁盘上

  • 阻塞挂起状态（Blocked-suspend)

  • 就绪挂起状态（Ready-suspend）

进程PCB管理-状态队列：

  • 由操作系统维护一系列队列

  • 不同状态的进程放到不同的队列中（根据优先级不同由多个就绪队列，根据阻塞原因不同有多个阻塞队列）

  • 当进程状态发生变化时，会从一个队列到另一个队列

7.9 什么是线程

线程的定义：

  • 进程当中的一条执行流程

进程的定义：

  • 原来的定义

  • 进程是一个资源平台，包括代码段，数据，打开的文件等

线程和进程的比较：

  • 进程是资源分配单位，线程是cpu调度单位

  • 进程有一个完整的资源平台，而线程只独享必不可少的资源，如寄存器，堆栈

  • 线程同样具有就绪，阻塞，执行三种基本状态

  • 线程能减少并发执行的时间和空间开销：

    • 线程的创建时间比进程短

    • 线程的终止时间比进程短

    • 同一进程内的线程切换时间比较短

    • 由于同一个进程的各线程共享内存和文件资源，不需要内核的通信

线程的优点：

  • 线程之间可以并发地执行

  • 线程之间共享相同的地址空间

线程的缺点：

  • 一个线程崩溃，会导致其所属进程的所有线程崩溃

    • 例如打开网页如果用多个线程，一旦一个网页有问题会导致所有网页关闭，所以像Chrome打开一个网页都是一个进程
